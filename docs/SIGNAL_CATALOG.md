# TrendScan Signal Catalog

> **Comprehensive inventory of every signal / metric / indicator computed by the TrendScan codebase.**
>
> Generated by reading every file under `src/lib/{board,regime,factors,scanner}/`,
> `scripts/compute_crypto_factors.js`, and `scripts/build_snapshot.js`.
>
> Each entry documents: **name · file · what it measures · inputs · output ·
> used by · in signal engine? · predictive potential**.

## How to read this document

- **In signal engine?** — Whether the calculation is wired into the Factor
  Monitor's stance engine (`compositeEngine.computeFactorStance`) or the
  Macro Regime's allocation engine (`regimeSignals.computeAllocation`).
  These are the two "verdict engines" that produce actionable labels
  (CONSTRUCTIVE / SELECTIVE / DEFENSIVE / WAIT, or ALLOCATE / STABLECOINS).
- **Predictive potential** — A forward-looking judgment:
  - **HIGH** → directly drives a trend-continuation / trend-reversal call
    (e.g. adaptive z of price, factor spread z, regime quadrant flip).
  - **MEDIUM** → confirms or contextualises a primary signal (e.g. RSI,
    crowding score, breadth pct).
  - **LOW** → display-only metric (e.g. sparkline, YTD return, n-count).

## Status of the `src/lib/signal/compute.js` 9-gate engine

**It does not exist in the codebase.** A search of the entire repo (`rg
"9 gates"`, `rg "compute.js"`, `rg "signal/compute"`) returned no matches.
The closest existing "verdict engine" is the **4-gate**
`compositeEngine.computeFactorStance` (stretch / persistence / crowding /
breadth) in `src/lib/factors/compositeEngine.js`, which produces
CONSTRUCTIVE / SELECTIVE / DEFENSIVE / WAIT labels with a 0–10 confidence.
The other verdict engine is `regimeSignals.computeAllocation`, which
combines Ultra6 + OB1 + Core9 to emit ALLOCATE / STABLECOINS with
MAXIMUM / HIGH / STANDARD conviction.

The /signal page (`src/pages/Signal.jsx`) also does **not** exist. The
site's routes today are `/` (Scanner), `/board`, `/macro`.

Any future `src/lib/signal/compute.js` should import from this catalog's
existing primitives (`adaptiveZ`, `horizonReturnWithStats`,
`computeFactorStance`, `computeRSI`, `computeOBV`, `computeMetrics`,
`computeFundingPremia` — see §11 below) rather than re-deriving them.

### 1.2 `scoreTheme(metrics, themeName)` — composite theme score (0–100)

**Measures**: weighted blend of breadth, leadership, momentum, relative strength.
**Inputs**: per-asset metrics from `computeMetrics` for all assets in a theme.
**Output**: `{score, pctAbove20, pctAbove50, pctAbove200, pctNewHigh,
pctNewHigh52w, avgRet20, avgRet5d, avgAtrExt, memberCount, avg_rs_btc_20d,
breadth, leadership, momentum}`.
**Weights**: `0.30*breadth + 0.25*leadership + 0.30*momentum + 0.15*rsComponent`
where:
- breadth = (pctAbove20 + pctAbove50) / 2
- leadership = pctNewHigh20 * 0.6 + pctAbove200 * 0.4
- momentum = scaleTo100(avgRet20, -0.10, 0.15)
- rsComponent = clamp(50 + avg_rs_btc_20d * 500, 0, 100)
**Used by**: `ThemesTab`, `DailyBoard`, `MomentumTab`, `buildThemeSectorRotation`.
**In signal engine?**: YES — feeds `themeStatus`.
**Predictive potential**: HIGH — broad theme leadership signal.

### 1.3 `themeStatus(score, delta, avgAtrExt)` — 8-state verdict label

**Measures**: categorical theme health (the only STRONG/WEAK/NEUTRAL-style
verdict in the current codebase).
**Output**: one of
`'STRONG / HOT' | 'DOMINANT' | 'EMERGING' | 'STRONG' | 'FADING' |
'WEAK' | 'IMPROVING' | 'DETERIORATING' | 'NEUTRAL'`.
**Logic**:
| Score | Delta | AvgAtrExt | Label |
|---|---|---|---|
| ≥75 | — | >6 | STRONG / HOT |
| ≥75 | — | — | DOMINANT |
| ≥60 | >2 | — | EMERGING |
| ≥60 | — | — | STRONG |
| ≤35 | <-2 | — | FADING |
| ≤35 | — | — | WEAK |
| — | >3 | — | IMPROVING |
| — | <-3 | — | DETERIORATING |
| — | — | — | NEUTRAL |

**Used by**: `ThemesTab` (status pill), `DailyBoard` (ThemeStatus summary).
**In signal engine?**: YES — this IS the verdict.
**Predictive potential**: HIGH — already produces STRONG/WEAK/NEUTRAL-style calls.

### 1.12 `buildRegimeLabel(regime)` — 4-state market regime

**Measures**: crypto-native risk regime from % assets above MA.
**Output**: `{label: 'RISK-ON'|'RISK-ON/NARROW'|'RISK-OFF'|'MIXED', color}`.
**Logic**:
- pctAbove50 ≥ 60 AND pctAbove200 ≥ 55 → RISK-ON (green)
- pctAbove50 ≥ 50 → RISK-ON/NARROW (blue)
- pctAbove50 ≤ 35 → RISK-OFF (red)
- else → MIXED (neutral)

**Used by**: `DailyBoard` (header).
**In signal engine?**: YES — direct verdict.
**Predictive potential**: HIGH.

### 1.13 `buildQuickView(rawResults, hlTickers)` — 5-bucket market summary

Five named buckets, each top-5 by an internal score:

| Bucket | Measures | Filter | Sort key |
|---|---|---|---|
| `strongest` | composite of ret20d (40%) + rs_btc_20d*100 (30%) + trendScore*5 (30%) | none | composite desc |
| `pickingUp` | recent acceleration | ret1w > 0 AND ret1w > ret20d/4 | (ret1w − ret20d/4) desc |
| `crowded` | long crowding via Hyperliquid funding | funding > 0.00005 AND crowdingScore > 10 | crowdingScore desc |
| `washedOut` | oversold candidates | ret60d < −0.15 | ret60d asc |
| `bigMoves` | largest |ret1d| | abs(ret1d) > 0.03 | abs(ret1d) desc |

**Used by**: `QuickViewBar` (top of Board).
**In signal engine?**: YES — `crowded` bucket annualises funding (`*3*365*100`)
  which is the only place funding rate currently becomes a tradeable signal
  in the codebase.
**Predictive potential**: HIGH — `crowded` is the closest existing
  funding-based signal; `washedOut` is mean-reversion; `strongest` is trend.

## 3. Regime calculations

**File**: `src/lib/regime/regimeCalculations.js` (292 lines)
**Used by**: every regime + factors + scanner/factorEngine file. These are
the **statistical primitives** the whole site is built on.

### 3.1 `mean(arr)`, `stddev(arr, mu?)`, `sma(series, period)`, `rollingStd(series, period)`

Standard statistical helpers. Output: number.

### 3.2 `adaptiveZ(series, shortLen=90, longLen=365, shortWeight=0.60)` — **the core z-score**

**Measures**: where the current value sits relative to its own recent
vs long-term history. Blends short-term responsiveness with long-term structure.
**Inputs**: numeric series, short/long lookback windows, short weight.
**Output**: z-score (typically −3 to +3, clamped to ±3 in some callers).
**Formula**: `shortWeight * shortZ + (1-shortWeight) * longZ`.
**Used by**: every growth/inflation/liquidity signal in §2.
**In signal engine?**: YES — every composite signal starts here.
**Predictive potential**: HIGH — the canonical "is this unusual?" test.

### 3.3 `pctROC(series, n=13)`, `yoyROC(series, n=365)`, `pointChange(series, n=13)`

Rate-of-change helpers. Output: number (pctROC returns %).

### 3.4 `ema(series, period)`, `currentEMA(series, period)`

Standard EMA. Output: array (ema) or number (currentEMA).

### 3.5 `zToScore(z)` — z-score → 0-100 scale

**Formula**: `clamp(50 + z*10, 0, 100)`.
**Used by**: `computeNowcast`, gauge UIs.

### 3.6 `impulseZ(compositeSeries, deltaPeriod=13, normPeriod=52)` — speed of change

**Measures**: how fast the composite z is changing relative to its own
historical deltas.
**Output**: z-score clamped to ±3.
**Used by**: `computeNowcast` (50% weight).
**Predictive potential**: HIGH — leading indicator of regime shifts.

### 3.7 `weightedComposite(signals)` — weighted-blend z-score

**Measures**: weighted average of multiple z-scored signals, each clamped to ±3.
**Inputs**: `Array<{value, weight}>`.
**Output**: composite z-score.
**Used by**: every `computeXxxSignals()` to roll up individual signals.
**Predictive potential**: HIGH.

### 3.8 `computeNowcast(compositeHistory)` — regime nowcast score

**Measures**: 50/50 blend of current z (`meZ`) and `impulseZ`.
**Output**: `{meZ, impulseZ, nowcast (0-100), score (0-100)}`.
**Used by**: `MacroRegime` for growth/inflation/liquidity gauges.
**In signal engine?**: YES — feeds quadrant + liquidity classification.
**Predictive potential**: HIGH.

### 3.9 `classifyQuadrant(growNowcast, inflNowcast)` — 5-state macro quadrant

**Output**:
- gUp + iDown → GOLDILOCKS
- gUp + iUp → OVERHEAT
- gDown + iUp → STAGFLATION
- gDown + iDown → CONTRACTION
- else → TRANSITIONAL

**Threshold**: MIDLINE=50, BAND=5.
**In signal engine?**: YES — drives Ultra6 U1.
**Predictive potential**: HIGH.

### 3.10 `classifyLiquidity(liqNowcast)` — 3-state liquidity

**Output**: `'LOOSE' | 'NEUTRAL' | 'TIGHT'` (band ±5 around 50).

### 3.11 `classifyRegime(growNowcast, inflNowcast, liqNowcast)` — combined label

**Output**: `{quadrant, liquidity, label: 'QUADRANT + LIQUIDITY', key}`.

### 3.12 `computeGrandComposite(growNowcast, inflNowcast, liqNowcast)`

**Weights**: 0.33 / 0.33 / 0.34.
**Output**: number 0–100.
**Used by**: `RegimeCard` headline.

### 3.13 `computeRSI(series, period=14)` — Wilder's RSI

**Output**: 0–100 (or 50 if insufficient data).
**Used by**: `computeOB1Signals` (ETH RSI), `Scanner` RSI filter.
**In signal engine?**: YES (in OB1) — partial.
**Predictive potential**: HIGH — classic overbought/oversold.

### 3.14 `computeOBV(prices, volumes)` — On-Balance Volume

**Output**: cumulative volume series.
**Used by**: `computeOB1Signals`.
**Predictive potential**: HIGH — divergence vs price is a classic reversal signal.

### 3.15 `getNextExecutionDate()` — first Friday of next month

Utility for the Allocation panel. Not a signal.

### 3.16 `SEASON_LABELS`, `REGIME_COLORS`, `REGIME_BG_OPACITY`

Display constants. Map quadrant → season (SPRING/SUMMER/FALL/WINTER/FLUX).

## 4. Regime percentile

**File**: `src/lib/regime/regimePercentile.js` (126 lines)
**Used by**: `factorEngine.computeSpreadMonitor`, all factor spread z+%
calculations. Borrowed from factorwatch.ai methodology.

### 4.1 `adaptiveZWithPctile(series, shortLen=90, longLen=365, lookback=252)`

**Measures**: adaptive z-score + non-parametric percentile against trailing
252 overlapping windows.
**Output**: `{z, pctile (0-100)}`.
**Why**: z-scores assume normality; financial returns are fat-tailed. Pctile
keeps the z honest.
**In signal engine?**: YES — feeds factor spread monitor.
**Predictive potential**: HIGH — non-parametric extremes are rarer than
z=2 in calm regimes.

### 4.2 `percentileOf(current, history)` — simple rank

**Output**: 0–100 (50 = median).

### 4.3 `horizonReturn(prices, horizonDays)` — h-day compounded return

**Output**: decimal return (0.05 = +5%) or null.

### 4.4 `horizonReturnWithStats(prices, horizonDays, lookback=252)`

**Measures**: h-day return + z-score + percentile vs trailing overlapping windows.
**Output**: `{ret, z, pctile}` or null.
**Used by**: `factorEngine.computeSpreadMonitor` for 1d/5d/20d/60d horizons.
**In signal engine?**: YES — primary input to `computeFactorStance`.
**Predictive potential**: HIGH — multi-horizon extreme detection.

## 10. Factors engine

**File**: `src/lib/factors/compositeEngine.js` (203 lines)
**Used by**: `factorSignals.computeFactorSignals`, `FactorSignalCard`,
`FactorMonitor`. This is the **only existing 4-gate verdict engine** (the
closest analogue to the not-yet-existing /signal "9-gate" engine).

### 10.1 `computeFactorStance({spreadZ, spreadPctile, rotation, crowdingScore, confirmation, factorName})`

**Measures**: turns factor z-score + rotation + crowding + breadth into a
CONSTRUCTIVE / SELECTIVE / DEFENSIVE / WAIT verdict with 0–10 confidence.

**Gates** (4 total, not 9):

| # | Gate | Threshold | Logic |
|---|---|---|---|
| 1 | stretch | \|z\| ≥ 2.0 | significant move vs history |
| 2 | persistence | `rotation.confirmed === true` | 3-session-confirm rule |
| 3 | crowding | `crowdingScore < 0.7` | not too correlated with other factors |
| 4 | breadth | `confirmation >= 0.5` | >50% of quintile members agreeing |

**Verdict matrix**:

| Stretch | Persistence | Crowding | Breadth | Stance | Confidence |
|---|---|---|---|---|---|
| ✓ | ✓ | ✓ | ✓ | CONSTRUCTIVE | 9 |
| ✓ | ✓ | ✓ | ✗ | CONSTRUCTIVE | 7 |
| ✓ | ✓ | ✗ | — | SELECTIVE | 5 |
| ✓ | ✗ | — | — | WAIT | 3 |
| ✗ | ✓ | — | — | SELECTIVE | 5 |
| ✓ (negative z) | — | — | — | DEFENSIVE | 6 |
| ✗ | ✗ | — | — | WAIT | 2 |

**Cap**: crowded + CONSTRUCTIVE → SELECTIVE, confidence ≤5.

**Output**: `{stance, confidence (0-10), rationale: string[], color, gates,
raw: {spreadZ, spreadPctile, crowdingScore, confirmation, passCount}}`.

**Used by**: `factorSignals`, `FactorMonitor`, `FactorSignalCard`.
**In signal engine?**: YES — this IS the verdict engine.
**Predictive potential**: HIGH — the primary "should I act on this factor"
signal on the site.

### 10.2 `computeAllStances(spreadMonitorRows, rotation, crowdingMatrix)`

Runs `computeFactorStance` for every factor in the spread monitor, returns
the highest-confidence one as `primary` plus all stances sorted by confidence.

### 10.3 Constants

- `STRETCH_THRESHOLD = 2.0`
- `CROWDING_THRESHOLD = 0.7`
- `CONFIRMATION_THRESHOLD = 0.5`
- `STANCE`, `STANCE_COLORS` enums.

## 14. Scanner factor engine

**File**: `src/lib/scanner/factorEngine.js` (463 lines)
**Used by**: `compute_crypto_factors.js` (server-side), `FactorMonitor`
(client-side).

### 14.1 `computeFactorScores(universe)` — 5-factor scorecard per asset

| Factor | Formula | Analog |
|---|---|---|
| `momentum` (12-1mo) | `P(t-21d) / P(t-252d) - 1` (clamped [−0.95, 5.0]) | Momentum |
| `size` | `-log(marketCap)` (small = high score) | Size |
| `volatility` | `-stddev(daily returns, 30d)` (low vol = high score) | Low Vol |
| `beta` | `1 - OLS β vs BTC over 90d` (low beta = high score) | Low Beta |
| `liquidity` | `log(30d USD volume / marketCap)` (clamped [−10, 5]) | Liquidity |

**Inputs**: `Array<{symbol, candles, marketCap}>`.
**Output**: `Array<{symbol, scores: {momentum, size, volatility, beta, liquidity}}>`.

### 14.2 `buildQuintilePortfolios(scoredUniverse, factorName)`

Winsorize (2.5/97.5 pct) → z-score → sort high→low → split into 5 quintiles.
**Output**: `{quintiles: [Q1..Q5], longOnly: Q5, shortOnly: Q1, spread: Q5+Q1}`.

### 14.3 `computeSpreadMonitor(portfoliosByFactor, candlesBySymbol, benchmarkUniverse)`

For each factor: compute equal-weighted Q5, Q1, benchmark series; compute
relative (Q5 − benchmark) and spread (Q5 − Q1) returns at 1d/5d/20d/60d +
YTD; each with `{ret, z, pctile}` via `horizonReturnWithStats`.
**Output**: `{factor: {factor, label, rel_1d, spread_1d, rel_5d, spread_5d,
rel_20d, spread_20d, rel_60d, spread_60d, rel_ytd, spread_ytd}}`.
**Used by**: `compute_crypto_factors`, `FactorMonitor`, `compositeEngine`.
**In signal engine?**: YES — `spread_20d.z` is the primary input to
`computeFactorStance`.
**Predictive potential**: HIGH.

### 14.4 `detectFactorRotation(portfoliosByFactor, candlesBySymbol)` — snapshot leader

Snapshot-only: returns leader by 20d return + trailing 20d returns per
factor. Full rotation detection happens in `rotationDetector.detectRotation`
on the persisted history.
**Output**: `{leader_20d, leader_held_days: 0, flipped_from: null,
flip_flag: false, trailing_20d_returns: {...}}`.

### 14.5 `buildQuilt(portfoliosByFactor, candlesBySymbol)` — 13-month performance quilt

**Output**: `Array<{month: 'YYYY-MM', ranking: [{factor, label, return}]}>`.
**Used by**: `FactorQuilt` component.
**Predictive potential**: LOW (display).

### 14.6 Internal helpers

- `winsorize(arr, lower=0.025, upper=0.975)` — clip to percentile bounds
- `zScore(arr)` — mean/stddev normalisation
- `computeBeta(assetReturns, marketReturns)` — OLS β
- `buildEqualWeightSeries(symbols, candlesBySymbol)` — equal-weighted normalized price series
- `subtractSeries(a, b)` — normalised difference a_norm − b_norm
- `detectIdenticalQuintiles(portfoliosByFactor)` — diagnostic warning if two factors produce identical Q5
- `computeMonthlyReturns(priceSeries)` — approximate monthly returns (uses ~30 candles/month heuristic)

## 18. Recommended new signals to feed a /signal STRONG/WEAK/NEUTRAL engine

Based on the catalog above, here's what a hypothetical
`src/lib/signal/compute.js` (9 gates → STRONG/WEAK/NEUTRAL) should
combine. All inputs already exist in the codebase; nothing new needs to
be invented.

### Suggested 9 gates

| # | Gate | Source | Logic | Why |
|---|---|---|---|---|
| 1 | **Trend alignment** | `boardEngine.computeMetrics` | `above20 + above50 + above200 ≥ 2` | Multi-MA confirmation filters chop |
| 2 | **Momentum stretch** | `regimePercentile.horizonReturnWithStats(closes, 20)` | `z ≥ 1.5 OR z ≤ -1.5` | 20d return at multi-month extreme |
| 3 | **Volume confirmation** | `scanEngine.computeRVol(candles, 20)` | `rVol > 1.2 in trend direction` | Breakout without volume = fakeout |
| 4 | **RSI not exhausted** | `regimeCalculations.computeRSI(closes, 14)` | `40 ≤ RSI ≤ 75 for longs, 25 ≤ RSI ≤ 60 for shorts` | Avoid chasing parabolic extensions |
| 5 | **Funding not crowded** | `data/historical/{SYMBOL}/funding.json` (NEW — from `scripts/signal/fetch_data.js`) | `8h funding rate < 0.05% (annualised < 55%)` | Crowded longs get liquidated |
| 6 | **Funding not collapsing** | same | `8h funding rate > -0.02%` | Shorts paying too much = squeeze risk |
| 7 | **Relative strength vs BTC** | `boardEngine.computeMetrics → rs_btc_20d` | `rs_btc_20d > 0 for longs` | Don't fight BTC leadership |
| 8 | **ATR extension reasonable** | `boardEngine.computeMetrics → atrExt50ma` | `1 ≤ atrExt50ma ≤ 4` | Not yet stretched, not dead |
| 9 | **Breadth confirmation** | `boardEngine.buildBreadthSeries → dailySeries[last].adDiff` | `adDiff > 0 for longs` | Market-wide confirmation |

### Verdict mapping

| Pass count | Verdict |
|---|---|
| 7–9 gates pass | **STRONG** |
| 4–6 gates pass | **NEUTRAL** |
| 0–3 gates pass | **WEAK** |

### Why these 9

- Gates 1–4 reuse the existing Scanner + Board primitives — zero new
  math, just composition.
- Gates 5–6 are the new piece that the historical fetcher enables —
  previously the only funding signal was the live Hyperliquid snapshot in
  `buildQuickView.crowded`, which is point-in-time only.
- Gate 7 ensures alignment with BTC leadership (the strongest crypto
  regime signal in §2.1 G1 + §6.1 computeRegime).
- Gate 8 (ATR extension) is already used in `themeStatus` STRONG/HOT
  detection (score≥75 + atrExt>6).
- Gate 9 (breadth) ties the per-asset signal back to the market-wide
  regime label from `buildRegimeLabel`.

### Inputs required from `scripts/signal/fetch_data.js`

The historical fetcher (created in Task 1) provides the multi-year
klines + funding needed to:
1. Compute per-asset `adaptiveZWithPctile` of funding rate over 252
   windows (Gate 5 in percentile form, not just threshold).
2. Backtest the 9-gate engine across the 2022 bear, 2023 recovery, 2024
   halving rally, and 2025 distribution — every regime cycle.
3. Compute per-symbol "funding exhaustion" baselines (TIA's 4h funding
   cadence ≠ BTC's 8h cadence — needs per-symbol normalisation, which the
   13-symbol universe enables).

### Suggested file structure

```
src/lib/signal/
├── compute.js          # 9-gate verdict engine (NEW — see §15 above)
├── gates.js            # individual gate functions (NEW)
├── historicalLoader.js # loads data/historical/{SYMBOL}/*.json (NEW)
└── verdict.js          # STRONG/WEAK/NEUTRAL label mapping (NEW)
```

Each gate function should accept `{candles, funding, metrics, breadth,
btcMetrics}` and return `{pass: boolean, value: number, threshold: number,
rationale: string}` — same shape as `compositeEngine.computeFactorStance`
gates, just 9 of them.
